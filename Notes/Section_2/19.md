# Detailed Notes on Advisors in Spring AI

## 1. What Are Advisors?

### Definition
- **Advisors** = Interceptors or middleware for the prompt flow in Spring AI.
- Analogous to **filters** in traditional Spring web applications.
- Intercept **requests going to** and **responses coming from** LLM models.

### Core Purpose
- Apply **pre-processing** logic before a prompt reaches the LLM.
- Apply **post-processing** logic on the LLM's response.
- Handle **cross-cutting concerns** and **housekeeping activities** without touching core business logic.

---

## 2. What Advisors Enable

| Capability | Description |
|-----------|-------------|
| **Pre/Post Processing** | Execute logic before and after LLM communication |
| **Custom Logging** | Log every prompt and response |
| **Auditing** | Track and record all AI interactions |
| **Inject Behavior** | Add functionality without modifying core logic |
| **Chain Multiple Behaviors** | Define and sequence multiple advisors |
| **Cross-Cutting Concerns** | Security, logging, monitoring, rate limiting |

---

## 3. Request-Response Flow with Advisors

```
End User Request
      ↓
  REST API
      ↓
  ChatClient Bean
      ↓
[Advisor 1 - Pre-processing]
      ↓
[Advisor 2 - Pre-processing]
      ↓
    LLM Model
      ↓
[Advisor 2 - Post-processing]
      ↓
[Advisor 1 - Post-processing]
      ↓
  End User Response
```

- Advisors execute in **chain order**.
- Any number of advisors can be configured.
- Both request and response pass through all advisors.

---

## 4. Best Practices

| Practice | Explanation |
|---------|-------------|
| **Keep stateless** | Don't retain data across requests or sessions |
| **Single responsibility** | One advisor per concern (logging advisor, security advisor separately) |
| **Don't alter prompt meaning** | Unless explicitly required by business; defeats the purpose |
| **No core business logic** | Business logic belongs in controllers/services, not advisors |
| **Chain cleanly** | Use multiple focused advisors rather than one large advisor |

---

## 5. How to Configure Advisors

### Two Configuration Levels

#### Level 1: ChatClient Bean Level (Default Advisor)
- Applied to **all requests** through that ChatClient.
- Configured via `ChatClient.Builder.defaultAdvisors()`.

```java name=ChatClientConfig.java
@Bean
public ChatClient chatClient(ChatClient.Builder builder) {
    return builder
        .defaultAdvisors(new SimpleLoggerAdvisor())
        .build();
}
```

#### Level 2: Per-Request Level
- Applied to a **specific request** only.
- Configured via `.advisors()` on the prompt chain.

```java name=ChatController.java
return chatClient.prompt()
    .system(systemMessage)
    .user(userMessage)
    .advisors(new SimpleLoggerAdvisor())
    .call()
    .content();
```

---

## 6. Built-in Advisors

| Advisor | Purpose |
|---------|---------|
| `SimpleLoggerAdvisor` | Logs all requests and responses |
| `SafeguardAdvisor` | Blocks requests containing sensitive words |
| `PromptChatMemoryAdvisor` | Manages chat history/memory across conversations |

- New advisors added with each Spring AI version.
- Custom advisors can also be created.

---

## 7. Creating Custom Advisors

- Implement one or both interfaces:
    - `CallAdvisor` – synchronous (blocking) calls.
    - `StreamAdvisor` – streaming responses.
- Implementing both = supports both normal and streaming LLM calls.

---

## 8. Summary

| Concept | Key Point |
|---------|-----------|
| Advisors | Interceptors for LLM prompt/response flow |
| Similar To | Filters in traditional Spring web apps |
| Flow | Request → Advisors (pre) → LLM → Advisors (post) → Response |
| Configuration | Default (all requests) or per-request |
| Built-in | SimpleLoggerAdvisor, SafeguardAdvisor, PromptChatMemoryAdvisor |
| Custom | Implement CallAdvisor / StreamAdvisor |
| Best For | Logging, auditing, security, cross-cutting concerns |
| Avoid | Altering prompt meaning unnecessarily; core logic in advisors |
