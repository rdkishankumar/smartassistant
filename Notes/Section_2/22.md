# Detailed Notes on ChatOptions in Spring AI

## 1. What Are ChatOptions?

### Definition
- **ChatOptions** = Configuration settings that customize how an LLM behaves during chat/completion calls.
- Think of it as a **tuning panel** for the AI model.
- Allows developers to control creativity, randomness, verbosity, response length, model selection, and more.

---

## 2. Important ChatOptions Parameters

### 2.1 Model
- Select a **specific LLM model** from the provider.
- Different models excel at different tasks:
    - Text generation, reasoning → GPT-4o, Claude Sonnet.
    - Image generation → DALL-E, Stable Diffusion.
- Override the framework's default model selection.

### 2.2 frequencyPenalty
- **Range**: -2.0 to 2.0
- **Higher value** = Less repetition in responses.
- **Lower value** = More repetition (e.g., "the cat is sleeping, the cat is walking, the cat is playing").
- **Tip**: Find a sweet spot; don't always use extreme values.

### 2.3 presencePenalty
- **Range**: -2.0 to 2.0
- **Higher value** = Encourages LLM to introduce new words/topics.
- Controls topic diversity in responses.

### 2.4 temperature ⭐ (Most Used)
- **Range**: 0.0 to 1.0
- Controls **creativity and randomness**.
- `0.0` = Focused, deterministic, same answer every time (feels robotic).
- `1.0` = Highly creative, random, varied responses.
- **Recommended**: 0.7–0.8 for natural, human-like responses.
- ⚠️ Don't use with `topP` simultaneously.

### 2.5 topP
- Controls **randomness** via probability distribution.
- Example: If `topP = 0.9`, LLM only considers words with ≥90% probability.
- If `topP = 0.5`, LLM considers words with ≥50% probability (more varied).
- **Higher topP** = More randomness.
- ⚠️ Use either `temperature` OR `topP`, not both.

### 2.6 topK
- Limits the **number of candidate words** considered for each next token.
- Example: `topK = 50` → LLM picks from top 50 potential next words.
- **Higher topK** = More randomness.
- Works alongside topP for fine-grained control.

### 2.7 stopSequence
- Defines a **word or sequence** that stops response generation.
- Example: Pass `"]}"` to stop after a JSON array ends (prevents extra explanation).
- Useful for structured output (JSON, CSV) where you don't want LLM commentary.

### 2.8 maxTokens ⭐ (Most Used)
- Sets the **maximum number of tokens** in the response.
- Example: `maxTokens = 50` → Short, concise response (stops abruptly if needed).
- Acts as a **budget** for the response.
- Prevents unexpectedly long (costly) responses.

## 3. Quick Reference Table

| Option | Range | Effect | Common Use |
|--------|-------|--------|------------|
| `model` | Provider-specific | Selects LLM model | Switch between models |
| `frequencyPenalty` | -2.0 to 2.0 | Reduces repetition | Story/content generation |
| `presencePenalty` | -2.0 to 2.0 | Encourages new topics | Diverse responses |
| `temperature` | 0.0 to 1.0 | Controls creativity | General use (0.7–0.8 recommended) |
| `topP` | 0.0 to 1.0 | Probability-based randomness | Alternative to temperature |
| `topK` | Integer | Limits word candidates | Fine-grained randomness control |
| `stopSequence` | String/list | Stops at keyword | Structured output (JSON, etc.) |
| `maxTokens` | Integer | Limits response length | Cost control, concise answers |

## 4. How to Configure ChatOptions

### Option 1: Default Options (All Requests via ChatClient)
```java name=ChatClientConfig.java
@Bean
public ChatClient chatClient(ChatClient.Builder builder) {
    ChatOptions options = ChatOptions.builder()
        .model("gpt-4o-mini")
        .temperature(0.7)
        .maxTokens(500)
        .presencePenalty(0.5)
        .build();

    return builder
        .defaultOptions(options)
        .defaultAdvisors(new SimpleLoggerAdvisor())
        .build();
}
```

### Option 2: Per-Request Options (Specific REST API)
```java name=ChatController.java
return chatClient.prompt()
    .system(systemMessage)
    .user(userMessage)
    .options(ChatOptions.builder()
        .temperature(0.9)
        .maxTokens(200)
        .build())
    .call()
    .content();
```

### Comparison: Default vs. Per-Request
| Method | Scope | Use When |
|--------|-------|----------|
| `defaultOptions()` on builder | All requests | Common settings for all APIs |
| `options()` on prompt | Specific request | Different settings per API |

## 5. Generic vs. Provider-Specific Options

### Generic ChatOptions
- Common parameters supported by most LLM providers.
- Interface: `ChatOptions` (Spring AI).
- Works across OpenAI, Gemini, Ollama, HuggingFace, etc.

### Provider-Specific ChatOptions
- Each provider may have extra parameters unique to their models.
- Use provider-specific classes for those options.

| Provider | Specific Class |
|----------|---------------|
| OpenAI | `OpenAiChatOptions` |
| Ollama | `OllamaChatOptions` |
| Google Gemini | `VertexAiGeminiChatOptions` |
| AWS Bedrock | `BedrockChatOptions` |

```java name=ChatClientConfig.java
// Using OpenAI-specific options
OpenAiChatOptions openAiOptions = OpenAiChatOptions.builder()
    .model("gpt-4o")
    .temperature(0.8)
    .maxTokens(1000)
    // OpenAI-specific options here
    .build();
```

## 6. Best Practices

| Practice | Explanation |
|---------|-------------|
| **Use defaults wisely** | Spring AI's defaults are reasonable; only override if needed |
| **Don't use temperature + topP together** | They serve the same purpose; pick one |
| **Test different values** | Find the sweet spot for your use case |
| **Use maxTokens for cost control** | Prevents unexpectedly large bills |
| **Use temperature 0.7–0.8** | Best balance of creativity and consistency |
| **Use stopSequence for structured output** | Prevents unwanted commentary after JSON/CSV |

---

## 7. Summary

| Concept | Key Point |
|---------|-----------|
| ChatOptions | Tuning panel for LLM behavior (model, creativity, length, etc.) |
| temperature | Most important; controls creativity (0=focused, 1=random) |
| maxTokens | Limits response length; controls cost |
| topP vs. temperature | Use one or the other, not both |
| Generic ChatOptions | Works across all providers |
| Provider-Specific Options | Use for provider-exclusive features |
| Configuration Level | Default (all requests) or per-request |
