# Detailed Notes on Building a Custom Advisor in Spring AI

## 1. Use Case: Token Usage Audit Advisor

### Why Track Tokens?
- LLM vendors (e.g., OpenAI) charge **per token consumed** (input + output).
- Example OpenAI pricing (GPT-4o-mini):
    - Input: $0.15 per 1 million tokens.
    - Output: $0.60 per 1 million tokens.
- Tracking tokens helps:
    - Monitor costs per request.
    - Save usage data to database for billing users.
    - Optimize prompts to reduce costs.

---

## 2. Creating the Custom Advisor

### Package and Class
- Package: `advisors`
- Class: `TokenUsageAuditAdvisor`
- Implement: `CallAdvisor` (synchronous; `StreamAdvisor` covered later)

### Full Implementation
```java name=TokenUsageAuditAdvisor.java
package com.easybytes.springai.advisors;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.ai.chat.client.advisor.api.AdvisedRequest;
import org.springframework.ai.chat.client.advisor.api.AdvisedResponse;
import org.springframework.ai.chat.client.advisor.api.CallAdvisor;
import org.springframework.ai.chat.client.advisor.api.CallAroundAdvisorChain;
import org.springframework.ai.chat.model.ChatResponse;

public class TokenUsageAuditAdvisor implements CallAdvisor {

    private static final Logger logger = LoggerFactory.getLogger(TokenUsageAuditAdvisor.class);

    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request, CallAroundAdvisorChain chain) {
        // Forward request to LLM (no pre-processing needed for token logging)
        AdvisedResponse advisedResponse = chain.nextAroundCall(request);

        // Extract token usage from response
        ChatResponse chatResponse = advisedResponse.response();

        if (chatResponse != null && chatResponse.getMetadata() != null) {
            var usage = chatResponse.getMetadata().getUsage();
            if (usage != null) {
                logger.info("Token usage details: {}", usage.toString());
                // Individual fields available:
                // usage.getPromptTokens()      → input/request tokens
                // usage.getCompletionTokens()  → output/response tokens
                // usage.getTotalTokens()       → prompt + completion tokens
            }
        }

        return advisedResponse;
    }

    @Override
    public String getName() {
        return TokenUsageAuditAdvisor.class.getSimpleName();
    }

    @Override
    public int getOrder() {
        return 1;  // Higher than SimpleLoggerAdvisor (order=0); executed first
    }
}
```

---

## 3. Understanding Token Usage Fields

| Field | Description |
|-------|-------------|
| `getPromptTokens()` | Input/request tokens consumed |
| `getCompletionTokens()` | Output/response tokens generated |
| `getTotalTokens()` | Sum of prompt + completion tokens (total cost basis) |

- **Note**: Token details come from the **response** (decided by LLM after processing).
- No token data available in the request phase.
- Token structure is **vendor-specific** (OpenAI shown here; other vendors may differ).
- Local/self-hosted LLMs (Ollama, Docker): No token billing; may not include usage data.

---

## 4. Advisor Order and Execution

| Advisor | Order | Executed |
|---------|-------|---------|
| `TokenUsageAuditAdvisor` | `1` (higher) | **First** |
| `SimpleLoggerAdvisor` | `0` (lower) | **Second** |

- Higher order number = higher priority = executed first.
- Same order = undefined execution sequence.

## 5. Configuring the Custom Advisor

### Option 1: Per-Request (Not Recommended for Common Advisors)
```java name=ChatController.java
return chatClient.prompt()
    .system(systemMessage)
    .user(message)
    .advisors(new TokenUsageAuditAdvisor())  // only for this API
    .call()
    .content();
```

### Option 2: ChatClient Bean Level (Recommended)
```java name=ChatClientConfig.java
@Bean
public ChatClient chatClient(ChatClient.Builder builder) {
    return builder
        .defaultAdvisors(List.of(
            new SimpleLoggerAdvisor(),
            new TokenUsageAuditAdvisor()
        ))
        .build();
}
```

- Bean-level configuration applies to **all REST APIs** using this ChatClient.
- Recommended since token auditing is a common cross-cutting concern.

## 6. Testing and Observations

### Simple Chat Request
- Prompt: "How to reset my password?"
- Token usage example: Prompt=90, Completion=82, **Total=172 tokens**.

### Prompt Stuffing Request (More Context = More Tokens)
- Prompt: With HR policy context data.
- Token usage: **Total=284 tokens** (higher due to large system message).
- Confirms: More context data → More tokens → Higher cost.
- Reinforces: Don't use Prompt Stuffing for large data volumes.

| Use Case | Implementation |
|----------|---------------|
| **Cost Monitoring** | Log tokens per request to console or monitoring tools |
| **User Billing** | Save token usage to database; charge users based on consumption |
| **Budget Alerts** | Trigger alerts when token usage exceeds thresholds |
| **Cost Optimization** | Identify expensive prompts and optimize them |

## 8. Summary

| Concept | Key Point |
|---------|-----------|
| Custom Advisor | Implement `CallAdvisor` (sync) and/or `StreamAdvisor` (streaming) |
| Token Usage | Available in response metadata (`getMetadata().getUsage()`) |
| Token Fields | `getPromptTokens()`, `getCompletionTokens()`, `getTotalTokens()` |
| Order | Higher number = executed first in the chain |
| Configuration | Per-request `.advisors()` or global `.defaultAdvisors()` (recommended) |
| Vendor-Specific | Token data structure varies by LLM provider |
| Real-World Use | Save to DB for billing, monitoring, cost optimization |
