## 2. What Are Message Roles?

### Definition
- When interacting with LLMs (like ChatGPT), we send **prompts made up of messages**.
- Each message has a **role** — a label that tells the LLM:
  - **Who** is sending the message.
  - **What context** to apply when responding.
- Roles allow more meaningful, controlled, and accurate responses from the LLM.

### Why Roles Are Important
- Without roles, the LLM has no context about who is speaking or what behavior is expected.
- With roles, developers can:
  - Restrict LLM behavior to specific domains.
  - Maintain conversation history.
  - Expose tools/functions for the LLM to call.
  - Separate system-level instructions from user input.

---

## 3. The Four Common Message Roles

### 3.1 User Role (Default)
- **Who**: The end user or developer sending input.
- **Purpose**: Carries the actual prompt, question, or instruction.
- **Assigned automatically** when a plain string is passed to `prompt()`.
- **Examples**:
  - "Tell me a joke."
  - "What are the top 3 places to visit in Rome?"
  - "How many leaves can I take?"

### 3.2 System Role
- **Who**: The developer/application (not the end user).
- **Purpose**: Provides instructions to the LLM on **how to behave** during the conversation.
- Set once before the conversation starts (acts as a "behavior guide").
- **Examples**:
  - "Be formal and professional while responding."
  - "You are a friendly tour guide."
  - "You are an internal HR assistant. Only answer HR-related questions about leave policies, working hours, benefits, and code of conduct."
- **Power**: Restricts the LLM from answering out-of-scope questions, making it suitable for enterprise use cases.

### 3.3 Assistant Role
- **Who**: The LLM model itself.
- **Purpose**: Represents the **LLM's response** to user messages.
- Named "Assistant" because LLMs act as assistants to humans.
- Used in **chat history management** — past LLM responses are stored as assistant messages and sent back in subsequent requests to maintain context.
- **Example**: LLM's reply: "The top 3 places to visit in Rome are the Colosseum, Vatican City, and the Trevi Fountain."

### 3.4 Function/Tool Role
- **Who**: A backend function or tool.
- **Purpose**: Provides special instructions for **invoking functions** or **fetching data** from storage systems.
- Used in the advanced concept of **Function Calling** (covered in upcoming sections).
- Allows LLMs to call developer-defined functions to retrieve real-time or organization-specific data.
- **Example**: A function that fetches current stock prices or retrieves company HR policy documents.
- Note: "Function" and "Tool" are used interchangeably in Spring AI.

## 6. Provider Support for Message Roles

Not all LLMs support all message roles. Choose your LLM accordingly.

| Provider | User | System | Assistant | Function/Tool |
|----------|------|--------|-----------|---------------|
| **OpenAI** | ✅ | ✅ | ✅ | ✅ |
| **Anthropic** | ✅ | ✅ | ✅ | ❌ |
| **Mistral AI** | ✅ | ✅ | ✅ | ❌ |
| **Google Gemini** | ✅ | ❌ | ✅ (called "Model") | ❌ |

### Key Notes
- **OpenAI**: Pioneer of GenAI; supports all four roles.
- **Anthropic & Mistral**: Support User, System, Assistant but NOT Function role.
- **Google Gemini**: Only supports User and Assistant (called "Model") roles; no System or Function role support.
- As a developer, understand which roles your chosen LLM provider supports before building business logic.

---

## 7. Spring AI's Smart Role Handling

### Automatic Fallback
- Spring AI handles role incompatibilities intelligently.
- **Example**: If you send a System message to Google Gemini (which doesn't support it):
  - Spring AI **automatically merges** the System message content **into the User message**.
  - The LLM still receives the instructions, just packaged differently.
- This abstraction saves developers from manually handling provider-specific limitations.

## 8. Summary

| Concept | Key Point |
|---------|-----------|
| Message Roles | Labels on messages that give LLMs context on who's speaking and how to respond |
| User Role | End user's questions/prompts (default) |
| System Role | Developer instructions for LLM behavior (restriction, tone, domain) |
| Assistant Role | LLM's responses; used in chat history |
| Function Role | Triggers function calls for real-time/org-specific data |
| Provider Differences | Not all providers support all roles; check before building |
| Spring AI Fallback | Automatically handles unsupported roles (e.g., System → User for Gemini) |
