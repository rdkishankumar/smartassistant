# Notes on Message Roles in LLMs (Spring AI)

## Introduction
- Basic Spring AI apps send prompts and receive responses, but advanced concepts unlock full potential:
    - Message roles, Advisors, Prompt Templates, Chat History Management, RAG, Function Calling, MCP clients/servers.
- Focus: Message Roles – help LLMs understand context for better responses.

## Message Roles Overview
- Messages sent to LLMs have roles for context.
- Common Roles:
    - **User**: User's prompts/questions (e.g., "Tell me a joke").
    - **System**: Instructions on LLM behavior (e.g., "Be formal and professional").
    - **Assistant**: LLM's responses (LLMs act as assistants).
    - **Function**: Instructions for functions/tools (e.g., data fetching; part of Function Calling – covered later).

## Example
- **System**: "You are a friendly tour guide."
- **User**: "What are the top 3 places to visit in Rome?"
- **Assistant**: LLM's response (e.g., list of places).

## Analogy: Stage Play
- **System**: Director giving stage instructions (e.g., "Act like a professional chef").
- **User**: Audience asking a question (e.g., "How do I cook pasta?").
- **Assistant**: Actor replying (e.g., "Boil water and add pasta for 8 minutes").
- **Function**: Backstage helper fetching ingredients/recipes.

## Provider Support
- Varies by LLM:
    - **OpenAI**: Supports all (User, System, Assistant, Function).
    - **Anthropic, Mistral AI**: Supports User, System, Assistant (no Function).
    - **Google Gemini**: Supports User and Model (Assistant) roles only (no System or Function).
- Spring AI handles incompatibilities:
    - E.g., For Gemini (no System), combines System message into User message automatically.
