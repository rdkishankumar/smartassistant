## One-Minute Setup

### **Prerequisites Check**

```bash
# Verify Node.js is installed
node --version
# Expected output: v18.17.0 (or newer)

# Verify npm
npm --version
# Expected output: 9.8.1 (or newer)

# Verify npx
npx --version
# Expected output: 9.8.1 (or newer)
```

### **Start MCP Inspector**

```bash
# Single command to start everything
npx @modelcontextprotocol/inspector

# What happens:
# 1. Downloads inspector package (if not cached)
# 2. Starts local server
# 3. Opens browser tab automatically
# 4. Shows UI at http://localhost:5173
```

### **Stop MCP Inspector**

```bash
# In the terminal running inspector
Ctrl+C

# Then close the browser tab
```

## Inspector Commands

### **Command Syntax**

```bash
npx @modelcontextprotocol/inspector
```

| Part | Purpose | Required |
|------|---------|----------|
| `npx` | Node Package eXecutor | Yes |
| `@modelcontextprotocol` | Package scope | Yes |
| `inspector` | Package name | Yes |
| No arguments | Starts with defaults | Yes |

### **Advanced Options (if available)**

```bash
# Custom port
npx @modelcontextprotocol/inspector --port 5174

# Open specific URL
npx @modelcontextprotocol/inspector --url http://localhost:5174

# Verbose logging
npx @modelcontextprotocol/inspector --verbose
```

## Connection Checklist

### **Before Connecting to MCP Server**

```
□ Node.js installed (node --version)
□ MCP Inspector running (npx @modelcontextprotocol/inspector)
□ Browser opened (http://localhost:5173)
□ Transport type known (Stdio, HTTP, WebSocket, etc.)
□ Server command ready (npx ... or docker or node)
□ Arguments prepared (-y, package name, paths)
□ Environment variables collected (if needed)
□ Authentication tokens/keys ready (if needed)
```

### **Connecting to Filesystem Server**

```
Step 1: Select Transport
├─ Choose: Stdio

Step 2: Enter Command
├─ Command: npx
├─ Arguments: -y @modelcontextprotocol/server-filesystem /Users/eazybytes/mcp

Step 3: Add Environment (Skip if none)
├─ Leave empty for filesystem

Step 4: Click Connect
└─ Wait for green dot

Step 5: View Tools
└─ See: read_file, write_file, list_directory, etc.
```

### **Connecting to GitHub Server**

```
Step 1: Prepare Token
├─ GitHub Settings → Developer Settings → Tokens
├─ Create token with: repo, read:user, gist, workflow
├─ Copy token (shown once)

Step 2: Select Transport
├─ Choose: Stdio

Step 3: Enter Command
├─ Command: npx
├─ Arguments: -y @modelcontextprotocol/server-github

Step 4: Add Environment Variables
├─ GITHUB_PERSONAL_ACCESS_TOKEN = ghp_your_token_here

Step 5: Click Connect
└─ Wait for green dot

Step 6: View Tools
└─ See: create_repository, list_issues, create_pull_request, etc.
```

## Troubleshooting

### **Issue: "npx: command not found"**

```
Problem: npx not available
Solution:
1. Check Node.js installation: node --version
2. If not installed, go to nodejs.org
3. Download and install Node.js
4. Restart terminal
5. Try again: npx --version
```

### **Issue: Port Already in Use**

```
Problem: Port 5173 already in use
Solution:
1. Find process using port:
   macOS/Linux: lsof -i :5173
   Windows: netstat -ano | findstr :5173

2. Kill process:
   macOS/Linux: kill -9 <PID>
   Windows: taskkill /PID <PID> /F

3. Or use different port:
   npx @modelcontextprotocol/inspector --port 5174
```

### **Issue: "Cannot find module"**

```
Problem: Inspector package not found
Solution:
1. Clear npm cache: npm cache clean --force
2. Reinstall: npx @modelcontextprotocol/inspector
3. Ensure npm is updated: npm install -g npm
4. Try again
```

### **Issue: Cannot Connect to Server**

```
Problem: Connection fails after clicking Connect
Solution:
1. Verify command syntax (copy from documentation)
2. Check arguments are correct
3. Verify directory/path exists
4. For GitHub: verify token is valid and has right scopes
5. Try running command directly in terminal first:
   npx -y @modelcontextprotocol/server-filesystem /path

6. Check logs in terminal for error messages
```

### **Issue: "Secure your MCP Server" Warning**

```
Problem: Security warning when connecting
Context: This is normal for local/development use
Solution:
1. Acknowledge the warning (it's informational)
2. For production, implement proper security
3. For dev/testing, you can dismiss safely
```

## Common Use Cases

### **Use Case 1: Quick Server Exploration**

**Scenario:** Evaluating a new MCP server before integrating

```
Time: 5 minutes

Process:
1. Start inspector: npx @modelcontextprotocol/inspector
2. Connect to server (use their example command)
3. Review tools list
4. Click 2-3 tools to read descriptions
5. Test 1 tool with sample data
6. Close inspector: Ctrl+C

Result: Know exactly what server can do
```

### **Use Case 2: Debugging Failed Integration**

**Scenario:** Tool call fails in Spring AI app

```
Time: 10 minutes

Process:
1. Start inspector
2. Connect to same server
3. Test exact same tool with exact same params
4. If it works in inspector: issue is in LLM/Spring
5. If it fails in inspector: issue is server/tool
6. Make notes and fix accordingly

Result: Narrow down root cause quickly
```

### **Use Case 3: Testing New Tool Capability**

**Scenario:** Added new tool to custom MCP server

```
Time: 2 minutes

Process:
1. Start inspector
2. Connect to server
3. Scroll to find new tool
4. Click to see details
5. Test with sample inputs
6. Verify output format

Result: Confident tool works before LLM integration
```

### **Use Case 4: Learning MCP Protocol**

**Scenario:** Understanding how tools work

```
Time: 30 minutes

Process:
1. Start inspector
2. Connect to simple server (filesystem)
3. Test several different tools
4. Observe input/output patterns
5. Try different parameter combinations
6. See how errors are handled

Result: Deep understanding of MCP protocol
```

## Pro Tips

### **Efficiency Tips**

```
TIP 1: Copy Commands from Docs
└─ Don't type commands manually
   └─ Copy from official documentation
      └─ Paste into inspector
         └─ Fewer typos, faster

TIP 2: Test with Real Data
└─ Test with actual file paths/parameters
   └─ Same as LLM will use
      └─ Catches real issues

TIP 3: Document Tool Capabilities
└─ While in inspector, note what's NOT available
   └─ No delete_file tool? Document it
      └─ Share findings with team

TIP 4: Use Favicon for Quick Spot
└─ Leave inspector tab open
   └─ Easy to spot in browser tabs
      └─ Quick reference while coding

TIP 5: Test Error Cases
└─ Try invalid parameters
   └─ See how server handles errors
      └─ Prepare error handling code
```

### **Security Tips**

```
✅ DO:
├─ Use environment variables for secrets
├─ Never hardcode tokens in inspector UI
├─ Use tokens with minimal permissions
├─ Rotate tokens regularly
└─ Use different tokens for dev/test/prod

❌ DON'T:
├─ Paste secrets into notes
├─ Share screenshots with credentials
├─ Use same token everywhere
├─ Commit secrets to git
└─ Leave inspector running publicly
```
