````markdown name=MCP-TRANSPORTS-COMPLETE-GUIDE.md
## Transport Overview

### **What are MCP Transports?**

MCP Transports define **how** data flows between:
- **MCP Client** (in Spring AI application)
- **MCP Server** (capability provider)

They answer: **What channel/mechanism is used for communication?**

### **Current MCP Transports (As of 2024)**

```
MCP Protocol
├─ STDIO (Local processes)
├─ Streamable HTTP (Remote/internet)
└─ SSE (Deprecated - use Streamable HTTP instead)
```

## STDIO Transport

### **What is STDIO?**

**STDIO = Standard Input/Output**

- Communication between two **local processes** on the **same machine**
- Data flows via OS pipes, not network
- Like two programs talking directly through their input/output streams
- No HTTP, no internet, no browser needed

### **When to Use STDIO**

```
✅ USE STDIO WHEN:
├─ MCP Server and MCP Client on same machine
├─ Local development/testing
├─ Command-line tools
├─ Terminal-based applications
├─ Docker containers on same host
├─ Scripts and automation
└─ Simple prototyping
```

## Streamable HTTP Transport

### **What is Streamable HTTP?**

**Streamable HTTP = Internet-friendly MCP transport**

- Communication over **HTTP protocol** across networks
- Client sends requests via **HTTP POST**
- Server responds with data that can be **streamed** (optionally via SSE)
- Works over internet, cloud, mobile, anywhere HTTP works
- Session-aware with headers like `MCP-Session-Id`
- True two-way communication

### **When to Use Streamable HTTP**

```
✅ USE STREAMABLE HTTP WHEN:
├─ MCP Server on different machine
├─ Remote/cloud deployment
├─ Web or mobile applications
├─ Multiple clients connecting
├─ Multi-user scenarios
├─ Production systems
├─ Need session management
├─ Across different networks/continents
└─ Enterprise/scalable setup
```

## Deprecated SSE Transport

### **What was SSE?**

**SSE = Server-Sent Events (Deprecated)**

- **One-way communication:** Server → Client only
- Could send live updates (like stock prices, weather)
- No way for client to send data back
- No session tracking
- No multi-user support
- **DEPRECATED as of MCP 1.0**

### **Why SSE Was Deprecated**

```
Limitations:
❌ One-way only (server to client)
   - Client couldn't talk back
   - Not suitable for interactive LLMs

❌ No session management
   - All events for all clients
   - No per-user state
   - No authentication per session

❌ No concurrent clients
   - Not designed for multiple users
   - Everyone gets same updates
   - No client differentiation

❌ Limited use cases
   - Only good for live updates
   - Stock tickers, weather, scores
   - Not enough for modern AI needs

History Timeline:
2023: SSE Transport (standalone)
       - Used for one-way streaming
       - Streaming stock prices example
2024: Streamable HTTP introduced
       - Replaces SSE
       - SSE now just one component
       - Full two-way communication
2025: SSE likely removed entirely
       - Deprecated, not recommended
       - Use Streamable HTTP instead
```

### **SSE Use Case Example (Now Deprecated)**

```
Old SSE-only scenario:
┌─────────────────────────────────┐
│   Stock Price Server            │
│   (Continuously updates prices) │
├─────────────────────────────────┤
│ Amazon: $150.50 ──┐            │
│ Google: $120.30 ──┼──► Client 1 │
│ Tesla:  $180.20 ──┤            │
│ Apple:  $160.40 ──┤            │
│ Meta:   $340.10 ──┘            │
└─────────────────────────────────┘

Problems:
❌ Client can't request specific stocks
❌ Client can't stop updates
❌ Client can't send data back
❌ Multiple clients get all updates

Solution:
Use Streamable HTTP instead
Client can: POST to request specific stocks
Server can: Stream responses via SSE
Everyone happy, backward compatible
```

## Comparison & Decision Matrix

### **Feature Comparison Table**

| Feature | STDIO | Streamable HTTP | SSE (Deprecated) |
|---------|-------|-----------------|------------------|
| **Local processes** | ✅ Perfect | ✅ Works | ❌ N/A |

| **Concurrent** | ❌ Single | ✅ Multiple | ❌ Broadcast |
| **Network friendly** | ❌ No | ✅ Yes | ✅ (limited) |
| **Over internet** | ❌ No | ✅ Yes | ✅ (limited) |

### **Decision Tree**

```
START: Which transport should I use?
│
├─ Are both Client and Server on the SAME MACHINE?

│  ├─ YES

│  │     (Local, fast, simple)
│  │
│  └─ NO
│     └─ Go to next decision
│
├─ Is this a PRODUCTION system?
│  │
│  ├─ NO (Development/Testing)
│  │  └─ Can use either:

│  │     - Streamable HTTP for cloud testing
│  │
│  └─ YES (Production)

│  │     (STDIO can't do this)
│  │
│  └─ NO

│
├─ Is the Server REMOTE or in the CLOUD?
│  │
│  ├─ YES
│  │  └─ Use Streamable HTTP ✅
│  │     (STDIO requires same machine)
│  │
│  └─ NO
│     └─ STDIO works fine
│
├─ Do you need SESSION MANAGEMENT?
│  │
│  ├─ YES
│  │  └─ Use Streamable HTTP ✅
│  │     (Session IDs, per-user state)
│  │
│  └─ NO
│     └─ Either transport works
│
└─ FINAL DECISION:
   ├─ Local, single-user, simple: STDIO
   ├─ Remote, multi-user, production: Streamable HTTP
   ├─ Never use: SSE (deprecated)
   └─ When in doubt: Streamable HTTP (more capable)

/**
 * Spring Boot Application with Local MCP Server (STDIO)
 */
@SpringBootApplication
public class LocalMcpApplication {

    public static void main(String[] args) {
        SpringApplication.run(LocalMcpApplication.class, args);
    }
}

/**
 * MCP Client Service (STDIO Transport)
 */
@Service
public class StdioMcpClientService {

    private static final Logger logger = 
        LoggerFactory.getLogger(StdioMcpClientService.class);

    private Process mcpServerProcess;
    private BufferedWriter serverInput;
    private BufferedReader serverOutput;

    /**
     * Start local MCP Server process
     */
    @PostConstruct
    public void initializeServer() throws IOException {
        logger.info("Starting local MCP Server process");

        // Start MCP server as local process
        ProcessBuilder pb = new ProcessBuilder(
            "java", "-jar", "/path/to/mcp-server.jar"
        );

        // Redirect error stream
        pb.redirectError(ProcessBuilder.Redirect.INHERIT);

        mcpServerProcess = pb.start();

        // Get input/output streams for communication
        serverInput = new BufferedWriter(
            new OutputStreamWriter(mcpServerProcess.getOutputStream())
        );
        serverOutput = new BufferedReader(
            new InputStreamReader(mcpServerProcess.getInputStream())
        );

        logger.info("MCP Server started (PID: {})", mcpServerProcess.pid());
    }

    /**
     * Send request via STDIO and get response
     */
    public String invokeToolViaStdio(String toolName,
                                     Map<String, Object> params)
            throws IOException {

        try {
            // Create JSON-RPC request
            String request = createJsonRpcRequest(toolName, params);
            logger.debug("Sending STDIO request: {}", request);

            // Send via stdout
            serverInput.write(request);
            serverInput.newLine();
            serverInput.flush();

            // Read response via stdin
            String response = serverOutput.readLine();
            logger.debug("Received STDIO response: {}", response);

            return response;

        } catch (IOException e) {
            logger.error("STDIO communication failed", e);
            throw e;
        }
    }

    /**
     * List tools via STDIO
     */
    public List<McpToolDefinition> discoverToolsViaStdio()
            throws IOException {

        String request = createJsonRpcRequest("tools/list", new HashMap<>());

        serverInput.newLine();

        return parseToolsResponse(response);

     * Cleanup on shutdown

/**
 * Spring Boot Application with Remote MCP Server (Streamable HTTP)
 */
@Service
public class StreamableHttpMcpClientService {

    private static final Logger logger = 
        LoggerFactory.getLogger(StreamableHttpMcpClientService.class);

    private final RestTemplate restTemplate;
    private final String mcpServerUrl;
    private String sessionId;

    @Value("${mcp.server.url:https://mcp-server.example.com}")
    private String mcpServerBaseUrl;

    @Value("${mcp.api.key}")
    private String mcpApiKey;

    public StreamableHttpMcpClientService(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
        this.mcpServerUrl = mcpServerBaseUrl + "/mcp";
        this.sessionId = UUID.randomUUID().toString();
    }

    /**
     * Invoke tool via Streamable HTTP
     */
    public String invokeToolViaHttp(String toolName,
                                   Map<String, Object> params)
            throws RestClientException {

        try {
            logger.info("Invoking tool via HTTP: {} with session: {}",
                       toolName, sessionId);

            // Create HTTP request headers
            HttpHeaders headers = new HttpHeaders();
            headers.setContentType(MediaType.APPLICATION_JSON);
            headers.set("MCP-Session-Id", sessionId);
            headers.set("Authorization", "Bearer " + mcpApiKey);

            // Create request body
            McpToolInvocationRequest request = 
                new McpToolInvocationRequest(toolName, params);

            HttpEntity<McpToolInvocationRequest> entity = 
                new HttpEntity<>(request, headers);

            // Send HTTP POST request
            ResponseEntity<String> response = restTemplate.exchange(
                mcpServerUrl + "/tools/call",
                HttpMethod.POST,
                entity,
                String.class
            );

            logger.info("Tool invocation successful: {}", 
                       response.getStatusCode());

            return response.getBody();

        } catch (HttpClientErrorException e) {
            logger.error("HTTP Client Error: {}", e.getStatusCode(), e);
            throw e;
        } catch (HttpServerErrorException e) {
            logger.error("HTTP Server Error: {}", e.getStatusCode(), e);
            throw e;
        }
    }

    /**
     * Stream tool responses (using SSE)
     */
    public void streamToolResponse(String toolName,
                                   Map<String, Object> params,
                                   Consumer<String> onChunk)
            throws RestClientException {

        try {
            logger.info("Starting stream for tool: {}", toolName);

            // Create headers for streaming
            HttpHeaders headers = new HttpHeaders();
            headers.set("MCP-Session-Id", sessionId);
            headers.set("Authorization", "Bearer " + mcpApiKey);
            headers.setAccept(Collections.singletonList(
                MediaType.parseMediaType("text/event-stream")
            ));

            // Create request
            HttpEntity<?> entity = new HttpEntity<>(headers);

            // Use WebClient for streaming (Spring 5+)
            WebClient webClient = WebClient.create(mcpServerUrl);

            webClient.post()
                .uri("/tools/call")
                .headers(h -> h.addAll(headers))
                .retrieve()
                .bodyToFlux(String.class)
                .doOnNext(chunk -> {
                    logger.debug("Received chunk: {}", chunk);
                    onChunk.accept(chunk);
                })
                .doOnError(e -> logger.error("Stream error", e))
                .doOnComplete(() -> logger.info("Stream completed"))
                .blockLast();

        } catch (Exception e) {
            logger.error("Stream invocation failed", e);
            throw new RuntimeException("Stream failed", e);
        }
    }

    /**
     * Discover tools via HTTP
     */
    public List<McpToolDefinition> discoverToolsViaHttp() {
        try {
            logger.info("Discovering tools via HTTP");

            HttpHeaders headers = new HttpHeaders();
            headers.set("MCP-Session-Id", sessionId);
            headers.set("Authorization", "Bearer " + mcpApiKey);

            HttpEntity<?> entity = new HttpEntity<>(headers);

            ResponseEntity<McpDiscoveryResponse> response = 
                restTemplate.exchange(
                    mcpServerUrl + "/tools",
                    HttpMethod.GET,
                    entity,
                    McpDiscoveryResponse.class
                );

            if (response.getBody() != null) {
                logger.info("Discovered {} tools", 
                           response.getBody().getTools().size());
                return response.getBody().getTools();
            }

        } catch (Exception e) {
            logger.error("Tool discovery failed", e);
        }

        return new ArrayList<>();
    }
}

## Interview Questions

### **Q1: When should you use STDIO vs Streamable HTTP for MCP?**

**Answer:**

**Use STDIO when:**
- MCP Client and Server are on the **same machine**
- Local development, testing, CLI tools
- You need simplicity and speed
- Example: Developer's laptop with Spring AI app + local MCP server

**Use Streamable HTTP when:**
- MCP Server is **remote** (different machine/cloud)
- Multiple clients need to connect
- Production deployments
- Multi-user or session-aware scenarios
- Example: Cloud-hosted MCP server serving multiple Spring AI apps

**Decision:** When in doubt, use **Streamable HTTP** (more capable, standard).

---

### **Q2: Why is SSE (Server-Sent Events) deprecated?**

**Answer:**

SSE had critical limitations:

1. **One-way only:** Server → Client only (no client requests)
2. **No session management:** All clients receive same data
3. **No multi-user support:** Can't differentiate between users
4. **Limited use cases:** Only good for simple live updates (stock prices, weather)

**Why Streamable HTTP replaced it:**
- ✅ Bidirectional (client can send requests via HTTP POST)
- ✅ Session-aware (per-user tracking)
- ✅ Multi-user (concurrent clients)
- ✅ Still supports streaming (SSE as optional component)

**Backward compatibility:** SSE functionality is now part of Streamable HTTP, so existing code won't break.

---

### **Q3: How does Streamable HTTP provide streaming if it's based on HTTP POST?**

**Answer:**

Streamable HTTP uses a **layered approach**:

```
Layer 1: REQUEST (Client → Server)
└─ HTTP POST (traditional request-response)
   - Client sends tool invocation
   - Can request specific tools/data

Layer 2: RESPONSE STREAMING (Server → Client)
└─ Server-Sent Events (SSE) [optional]
   - Server streams back chunks of data
   - Real-time updates
   - Long-lived connection

Result:
- Client can request what it needs (HTTP POST)
- Server can stream back responses (SSE)
- Full two-way, interactive communication
- Better than SSE alone (which couldn't request)
```

---

### **Q4: In a Docker setup with two containers, which transport should you use?**

**Answer:**

**Depends on container setup:**

**Same Docker host:**
```
Option 1: STDIO Transport
- If containers share a volume
- If using docker socket
- Local communication, fast

Option 2: Streamable HTTP
- Use inter-container networking
- More explicit, easier debugging
- Better for multi-host setups
```

**Different Docker hosts:**
```
Must use: Streamable HTTP
- Cross-host communication
- Network-based
- Requires HTTP/TLS setup
```

**Best practice:** Even on same host, use **Streamable HTTP** for:
- Better error handling
- Clearer logging
- Production-like setup
- Future migration flexibility

---

### **Q5: What does the "MCP-Session-Id" header do in Streamable HTTP?**

**Answer:**

`MCP-Session-Id` header:

1. **Identifies the client session**
   ```
   MCP-Session-Id: session-abc123-xyz789
   ```

2. **Enables session tracking**
   - Server knows which client sent request
   - Can maintain per-client state
   - Enables multi-user scenarios

3. **Example flow:**
   ```
   Client A: POST /tools/call
             Header: MCP-Session-Id: session-A
             → Server routes to Session A data

   Client B: POST /tools/call
             Header: MCP-Session-Id: session-B
             → Server routes to Session B data

   Same MCP Server, different clients, different states
   ```

4. **Benefits:**
   - ✅ Multi-user support
   - ✅ User context isolation
   - ✅ Persistent sessions
   - ✅ Audit logging per session
````
