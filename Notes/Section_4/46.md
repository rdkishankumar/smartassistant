# Detailed Notes on Implementing Chat Memory in Spring AI

## 1. Setup: Configuring ChatMemory in ChatClient

### Dependencies Needed
- Spring AI auto-creates beans for:
    - `MessageWindowChatMemory` (implements `ChatMemory`)
    - `InMemoryChatMemoryRepository` (implements `ChatMemoryRepository`)
- Developer only needs to wire them together via advisors.

### Configuration Code
```java name=ChatClientConfig.java
@Bean
public ChatClient chatMemoryChatClient(
        ChatClient.Builder builder,
        ChatMemory chatMemory) {  // Auto-injected by Spring AI

    // Create MessageChatMemoryAdvisor using builder pattern
    Advisor memoryAdvisor = MessageChatMemoryAdvisor.builder(chatMemory)
            .build();

    // Create SimpleLoggerAdvisor for request/response logging
    Advisor loggerAdvisor = new SimpleLoggerAdvisor();

    return builder
            .defaultAdvisors(List.of(loggerAdvisor, memoryAdvisor))
            .build();
}
```

### Dependency Injection Chain (Auto-handled by Spring)
```
ChatMemoryRepository (InMemoryChatMemoryRepository)
          ↓ injected into
ChatMemory (MessageWindowChatMemory)
          ↓ injected into
Constructor → passed to MessageChatMemoryAdvisor.builder()
```

## 2. REST API for Chat Memory

```java name=ChatMemoryController.java
@RestController
@RequestMapping("/api")
public class ChatMemoryController {

    private final ChatClient chatMemoryChatClient;

    public ChatMemoryController(ChatClient chatMemoryChatClient) {
        this.chatMemoryChatClient = chatMemoryChatClient;
    }

    @GetMapping("/chat-memory")
    public String chatMemory(@RequestParam String message) {
        return chatMemoryChatClient.prompt()
                .user(message)
                .call()
                .content();
    }
}
```

## 3. How It Works Internally

### Flow: Pre-Processing (Before LLM Call)
```
User sends: "What is my name?"
       ↓
MessageChatMemoryAdvisor.before() intercepts
       ↓
Calls ChatMemory.get(conversationId)
       ↓
Fetches all historical messages from InMemoryChatMemoryRepository
       ↓
Injects historical messages as structured messages into prompt
       ↓
LLM receives: full conversation history + new question
```

### Flow: Post-Processing (After LLM Response)
```
LLM sends response
       ↓
MessageChatMemoryAdvisor.after() intercepts
       ↓
Calls ChatMemory.add(conversationId, newMessages)
       ↓
Saves new user message + LLM response to InMemoryChatMemoryRepository
       ↓
Response returned to user
```

## 5. The Problem: Default Conversation ID

### Issue Identified
- All messages stored under a **single default conversationId = "default"**.
- Different users' messages are mixed together.
- LLM confuses messages across users.

### Demo of the Problem
| Action | Result |
|--------|--------|
| User 1 sends: "My name is Madan" | Stored under `conversationId = "default"` |
| User 2 sends: "My name is John Doe" | Also stored under `conversationId = "default"` |
| User 1 asks: "What is my name?" | LLM responds: "John Doe" ❌ (Wrong!) |

### Debug Verification
- Breakpoint in `MessageChatMemoryAdvisor.before()` method.
- `conversationId` value = `"default"` (confirmed).
- All 8 messages (from both users) loaded together.
- Mix of User Messages and Assistant Messages for both users.

### Root Cause
- `CONVERSATION_ID` not set per user.
- Framework defaults to `"default"` when no ID provided.

## 6. Summary

| Concept | Key Point |
|---------|-----------|
| Bean injection | Spring AI auto-creates `ChatMemory` and `ChatMemoryRepository` beans |
| `MessageChatMemoryAdvisor` | Created via `builder(chatMemory).build()` |
| Advisor registration | Via `defaultAdvisors(List.of(...))` on ChatClient builder |
| Memory storage | Structured messages (User, System, Assistant roles preserved) |
| Default conversationId | `"default"` — same for ALL users (problematic) |
| Problem | All users share same conversation history |
| Solution | Set unique conversationId per user (next lecture) |
