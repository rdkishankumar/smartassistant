# Detailed Notes on Chat Memory in Spring AI

## 1. What Is Chat Memory?

### Definition
- **Chat Memory** = A Spring AI concept that makes LLMs behave like humans by **remembering past conversations**.
- Improves LLM responses by providing full conversation context.

### How It Works (The Trick)
1. End user starts a conversation.
2. Each message is **stored in a storage system**.
3. Stored messages are **appended to new messages** before sending to LLM.
4. LLM receives entire conversation history → gives contextually accurate responses.

### Analogy
- Same technique used by ChatGPT, Gemini, and other LLM wrappers to simulate ongoing conversation.

---

## 2. Core Interfaces

### Interface 1: `ChatMemory`
- **Defines**: Memory strategy — **what to store and how**.
- Default `CONVERSATION_ID`: `"default"` (single conversation).
- Real apps: Use unique identifiers (username, email, mobile number) to differentiate users.

#### Memory Strategies
| Strategy | Description |
|----------|-------------|
| Last N messages | Keep only the most recent N messages |
| Time window | Store messages within a specific time range |
| Token limit | Retain messages until token budget is reached |

#### Key Methods
| Method | Purpose |
|--------|---------|
| `add()` | Add messages to storage |
| `get()` | Retrieve messages by conversationId |
| `clear()` | Clear conversation history |

---

### Interface 2: `ChatMemoryRepository`
- **Defines**: Storage contract — **where to store messages** (how to read/write).
- Like a JPA repository interface with:
    - `findByConversationId()` – Fetch messages.
    - `saveAll()` – Save messages.
    - `delete()` – Remove messages.

---

## 3. Implementation Classes

### `MessageWindowChatMemory` (implements `ChatMemory`)
- **Purpose**: Maintains a sliding window of messages (retains last N messages).
- **Default max messages**: 20.
- When limit exceeded: **Older messages evicted**.
- **Special handling for System Messages**:
    - If a new `SystemMessage` is added, all previous `SystemMessage` instances are removed.
    - When evicting, `SystemMessages` are **preserved**; other messages evicted first.

#### Internal Flow (`add()` method)
```
1. Load existing messages via findByConversationId()
2. Call process() → check if within limit
3. If within limit: Add new message
4. If over limit: Evict older messages (preserve SystemMessages)
5. Save updated list via saveAll()
```

#### Internal Flow (`get()` method)
```
1. Call findByConversationId(conversationId)
2. Return all stored messages for that conversation
```

---

### `InMemoryChatMemoryRepository` (implements `ChatMemoryRepository`)
- **Storage**: `ConcurrentHashMap` (in-memory; thread-safe).
- **Persistence**: Lost on app restart.
- **Use case**: Development, testing, or small-scale apps.

```java name=InMemoryChatMemoryRepository.java
// Internal storage structure (simplified)
ConcurrentHashMap<String, List<Message>> store = new ConcurrentHashMap<>();

// Save
store.put(conversationId, messages);

// Read
store.get(conversationId);

// Delete
store.remove(conversationId);
```

---

### `JdbcChatMemoryRepository` (implements `ChatMemoryRepository`)
- **Storage**: Relational databases (MySQL, PostgreSQL, etc.).
- **Persistence**: Survives application restarts.
- **Use case**: Production enterprise applications.
- **Requires**: JDBC dependency added to the project.

---

## 4. Comparison: Storage Options

| Implementation | Storage | Persistent? | Use Case |
|---------------|---------|-------------|----------|
| `InMemoryChatMemoryRepository` | JVM Memory (HashMap) | ❌ Lost on restart | Dev/Testing |
| `JdbcChatMemoryRepository` | MySQL, PostgreSQL, etc. | ✅ Survives restart | Production |
| Custom Implementation | MongoDB, Redis, etc. | Depends | Specialized needs |

---

## 5. Why Limit Messages (Max 20)?

### The Reason
- Sending unlimited messages = Unlimited tokens sent to LLM per request.
- **Token cost** grows with every message added to history.
- LLM **context window limits** also restrict how much data can be sent.
- Solution: Keep only **last N messages** (20 by default) as a balance between context and cost.

## 6. CONVERSATION_ID — Key Concept

| Scenario | CONVERSATION_ID Value |
|---------|----------------------|
| Default (single user) | `"default"` |
| Multi-user (real apps) | Username, email, mobile number, session ID |

- Real enterprise apps: **Always use unique identifiers** to separate conversations per user.

## 8. Summary

| Concept | Key Point |
|---------|-----------|
| Chat Memory | Appends past messages to new prompts for LLM context |
| `ChatMemory` | Interface defining memory strategy (what/how to store) |
| `ChatMemoryRepository` | Interface defining storage operations (where to store) |
| `MessageWindowChatMemory` | Keeps last 20 messages; evicts old ones; preserves SystemMessages |

| Why limit messages | Token cost and LLM context window constraints |
