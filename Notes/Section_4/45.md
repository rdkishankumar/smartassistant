# Detailed Notes on Chat Memory Advisors in Spring AI

## 1. The Problem: Who Invokes ChatMemory Methods?

### The Gap
- `MessageWindowChatMemory` has methods: `add()`, `get()`, `clear()`.
- `InMemoryChatMemoryRepository` has: `findByConversationId()`, `saveAll()`, `delete()`.
- But **someone must invoke these methods** and inject the retrieved messages into the LLM request.

### The Answer: Memory Advisors
- Spring AI provides **three specialized Memory Advisors** that:
    1. Intercept requests/responses (like all advisors).
    2. Call `ChatMemory` methods to fetch historical messages.
    3. Inject those messages into the LLM prompt.
    4. Store new messages after LLM responds.

---

## 2. Three Chat Memory Advisors

### 2.1 `MessageChatMemoryAdvisor` ⭐ (Recommended for Most Cases)

#### How It Works
- Fetches messages from `ChatMemory` storage.
- Injects them as **structured messages** (preserving their original roles).
- Each message type maintains its role: User → User message, Assistant → Assistant message, System → System message.

#### Best For
- LLMs that support all message roles (User, System, Assistant).
- When you want LLM to see the **full structured chat history**.
- Simulates a **real chat log**.

#### Example (What Gets Injected)
```
[System Message]: "You are an HR assistant."
[User Message]: "How many leaves do I get?"
[Assistant Message]: "You get 18 days annually."
[User Message]: "Can I carry them forward?"
[Assistant Message]: "Yes, up to 8 days."
[User Message]: "What about medical leave?"   ← New question
```

---

### 2.2 `PromptChatMemoryAdvisor`

#### How It Works
- Fetches messages from `ChatMemory` storage.
- Converts **entire chat history into plain text** (ignores message roles).
- Appends the plain text summary to the **System message**.

#### Best For
- Simple LLMs that **don't support all message roles**.
- LLMs without separate User/Assistant role support.
- When a text summary of history is sufficient.

#### Example (What Gets Injected into System Message)
```
System Message:
"You are an HR assistant.

Previous conversation:
User asked: How many leaves do I get?
Response: You get 18 days annually.
User asked: Can I carry them forward?
Response: Yes, up to 8 days."
```

---

### 2.3 `VectorStoreChatMemoryAdvisor`

#### The Problem It Solves
- Long-term users (2-3 years) accumulate **thousands of chat messages**.
- Sending only last 20 may miss important historical context.
- User may ask questions related to conversations from 6 months or 1 year ago.

#### How It Works
- Stores all chat messages in a **Vector Database** (not MySQL/MongoDB).
- When user asks a question: Search vector DB using **natural language**.
- Vector DB returns **only the most relevant messages** (e.g., top 10 from thousands).
- Those relevant messages are sent to the LLM.

#### Key Advantage
- Maintains **unlimited chat history** in vector DB.
- Only sends **relevant subset** to LLM (not all history).
- Uses **embeddings** for semantic search (similar to LLM internals).

#### Best For
- Long-term, knowledge-based conversations.
- High-volume chat histories.
- Applications needing semantic search over history.

- Topics: Vector databases, embeddings, configuration, read/write operations.

## 3. Comparison of Memory Advisors

| Advisor | Message Format | Storage | Best For |
|---------|---------------|---------|----------|
| `MessageChatMemoryAdvisor` | Structured (preserves roles) | In-Memory/JDBC | Most enterprise apps; LLMs supporting all roles |
| `PromptChatMemoryAdvisor` | Plain text summary | In-Memory/JDBC | Simple LLMs; limited role support |
| `VectorStoreChatMemoryAdvisor` | Semantic search results | Vector Database | Long-term/high-volume conversations |

---

## 4. How Memory Advisors Work with ChatMemory (Flow)

```
User sends new message
        ↓
Memory Advisor intercepts (pre-processing)
        ↓
Calls ChatMemory.get(conversationId)
        ↓
ChatMemory calls ChatMemoryRepository.findByConversationId()
        ↓
Returns list of historical messages
        ↓
Advisor injects historical messages into LLM prompt
        ↓
LLM processes full context (history + new message)
        ↓
LLM generates response
        ↓
Memory Advisor intercepts (post-processing)
        ↓
Calls ChatMemory.add(conversationId, newMessages)
        ↓
Stores new user message + LLM response in storage
        ↓
Response returned to user
```

---

## 5. When to Use Each Advisor

| Scenario | Recommended Advisor |
|---------|-------------------|
| Standard enterprise chatbot (OpenAI, Claude) | `MessageChatMemoryAdvisor` |
| LLM with limited role support | `PromptChatMemoryAdvisor` |
| Long-term user history (months/years) | `VectorStoreChatMemoryAdvisor` |
| HR assistant, IT helpdesk, customer support | `MessageChatMemoryAdvisor` |
| RAG-based knowledge applications | `VectorStoreChatMemoryAdvisor` |

---

## 6. Summary

| Concept | Key Point |
|---------|-----------|
| Memory Advisors | Bridge between ChatMemory and ChatClient |
| `MessageChatMemoryAdvisor` | Structured messages; preserves roles; best for most cases |
| `PromptChatMemoryAdvisor` | Plain text history; for simple/limited LLMs |
| `VectorStoreChatMemoryAdvisor` | Semantic search from vector DB; for long-term history |
| Vector DB | Stores unlimited history; returns only relevant messages |
| Embeddings | Used by vector DB for semantic search |
